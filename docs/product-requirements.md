# AnyDocsMarked 产品需求基线

## 核心目标

AnyDocsMarked 让通用 AI Agent 能够基于具体案卷，进行可信、完整、可追溯、可复核的案件整理与使用。Wiki、OCR、检索和 MCP 都是实现手段，不是产品目标。

产品优先保护可信下限：宁可明确查不出，也不把推断、通用知识或错误识别结果冒充本案事实。

## 目标用户与核心任务

目标用户是处理大量案件材料的律师、法务和法官助理等法律工作者。他们需要把 PDF、Word、扫描件、表格、聊天记录等异构材料整理为可持续使用的案件工作区，并据此查询事实、梳理关系、制作简报或开展分析。

用户的核心交互只有两类：

1. **build**：交付案件材料，要求 Agent 整理案件。
2. **answer**：就本案提问，或要求基于本案制作简报、汇报、梳理与分析。

## 必须交付的产品能力

### 材料转换

- 批量处理常见办公文档、PDF、扫描件和图片。
- 保留来源身份，暴露转换失败、跳过和质量可疑状态。
- 不因质量问题静默修改、删除或伪造内容。
- 任何外部处理必须在用户明确同意后发生。

### 案件整理

- 围绕案件主体、法律关系、法律事实和时间线组织案件知识。
- 案件事实必须具有逐字证据锚点。
- 分析与来源直接记载的事实显式分离。
- 来源冲突、OCR 可疑和无法确认的信息必须标为存疑，不得静默取舍。
- 每份材料必须处于已使用、说明理由后跳过或尚未处置三种状态之一；尚未处置时不得宣称整理完成。

### 检索与问答

- 回答本案问题前必须检索本案材料。
- Wiki 综合路径与原文召回路径保持独立，以便交叉验证和发现遗漏。
- 每项本案事实都必须能回到来源文本；未找到时明确说明。
- 证据冲突时并列呈现，Agent 可提供标注为分析的倾向，但最终判断归用户。
- 可选检索能力不可用时，核心案件整理与基于 Wiki 的问答仍可降级运行。

## 产品原则

优先级依次为：证据忠实、逐字可追溯、材料完整、不确定性透明、隐私安全、用户可用、检索效率、智能程度。

系统使用确定性检查约束证据入口和交付出口；中间的阅读、关联、组织和分析由宿主 Agent 完成。宿主 Agent 是唯一负责整理和生成回答的推理中心；换实例判官只能做狭窄验证，不生成案件知识或答案，检索模块也不负责回答。

三个模块保持独立：Makeitdown 负责来源文本，Lawiki 负责案件语义组织与交付约束，RAG Retriever 负责独立召回。它们通过稳定的文件与命令契约协作，不共享内部实现。

## 非目标

- 通用行业知识抽取平台或任意 Schema 抽取框架。
- 独立的 LLM 应用、第二套问答模型或强制 OpenAI API。
- 自动裁判、自动生成最终法律意见或替代律师判断。
- 用固定抽取流水线或自动页面生成取代 Agent 的案件整理。
- 与现有检索和案件工具重复的知识数据库或 MCP 服务。
- 在没有真实下游需求时预建机器可读导出体系。

## 产品级验收

一个案件只有在以下条件同时满足时才算可用：

- 非技术用户能够通过自然语言启动整理并继续就本案工作。
- 每份材料都有明确处置结果，失败、跳过和可疑状态不会静默发生。
- 案件事实能逐字回溯至来源文本，并最终定位到原始材料。
- 金额、日期、姓名等关键内容不会被模型改写成来源事实。
- 问答前经过检索，回答中的本案事实均有有效证据锚点。
- 查不到时拒绝编造，冲突时不静默裁决。
- 可选模块失败时核心能力可明确降级。
- 未经明确同意时，案件数据不会发送到外部服务。
- 最终法律判断始终由人完成。

## 变更准入

任何新增能力必须说明：它解决哪个用户步骤的问题、增强哪项可信目标、为何不能深化现有模块、由确定性代码还是 Agent 负责、失败时如何降级，以及怎样通过真实案件场景验收。无法回答这些问题的技术能力不进入主路线。
