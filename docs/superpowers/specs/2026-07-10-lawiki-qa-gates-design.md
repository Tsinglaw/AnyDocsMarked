# lawiki 问答确定性闸门（QA Gates）设计

状态：设计已获用户批准，待实现。

## 背景与目标

lawiki 的确定性资产（lint 锚点校验、蕴含判官、闭世界铁规）目前只覆盖**建库时**（wiki ingest）。
**问答时**的约束——先检索后作答、每个事实挂逐字锚点、闭世界——全部是 `qa.md`/`SKILL.md`
里的 prose 指令，靠模型自觉执行。对照 open-notebook（其 Ask 用 LangGraph 固定拓扑把
"检索由代码执行"锁死，但引用忠实只有 prose 恳求、无事后校验），两个项目恰好在互补的
两层各自做了强制：

| | open-notebook | lawiki（现状） |
|---|---|---|
| 流程服从（先检索后答） | ✅ 代码拓扑强制 | ❌ prose |
| 引用存在性 + 逐字忠实 | ❌ 无校验 | ✅ lint（仅 wiki 层） |

本设计把 open-notebook 的**原理**（确定性两端）借进 skill 形态，不借其**形态**
（独立应用/固定管线）：**过程保持 agent 自由（LLMwiki 式人类思考：读 wiki、顺
wikilink、四情形分流），产物必过机器闸门**。

```
【前闸门·确定性】evidence 一条命令产出证据包（检索由代码执行）
        ↓
【中段·自由】agent 综合 wiki 结论 × 证据包，四情形分流（协议不变）
        ↓
【后闸门·确定性】回答草稿过 lint answer，0 违规才允许交付
```

设计立场：管不住"agent 想什么"，但管得住"agent 说出来的每条引用都真实、逐字、
且指向本案 `_md/`"。对第一性原则（宁可查不出，也不编造）而言，锁内容比锁流程更本质——
编造的引用过不了闸门，无锚点的"事实"也过不了；唯一逃逸路径是把内容包装成
`> [!note] 分析`，而那本来就是显式标记为"非事实"的。

不新增界面、不引入第三方依赖、不改三层架构、不改 rag-retriever 本体。

## 1. 前置修复：面包屑默认锚点 bug（`tools/rag.py`）

**现象（2026-07-10 实测确认）**：rag-retriever 结构感知分块（2026-06-28）给存储的
chunk 文本加了标题面包屑前缀（`pipeline._compose`：`"民事判决书 > 本院认为\n\n正文"`），
让向量携带章节语境。`rag.py` 的 `enrich_hit` 拿带前缀的 `text` 拼默认锚点，而面包屑
在源文件里不是连续文本（两个标题之间隔着正文），归一化后无法定位：

```
✗ [片段不符] 找不到片段: 「民事判决书 > 本院认为 本院认为，被告应向原告偿还…」
```

**根因**：面包屑功能晚于 wrapper（2026-06-21）落地；`test_rag.py` 的"锚点必过真实
lint"测试只用了无面包屑的命中，没盖住新情况。wrapper 文档"片段经归一化后必能在
源文件定位"的承诺已失效。

**影响面**：ingest 场景失败得安全（写进 wiki 会被 lint 抓住）；问答场景没有 lint
兜底，agent 直引默认 `anchor` 会给出定位不到的引用——恰好伤在问答可溯源上。
本设计的 answer 闸门会让锚点用量大增，带病的默认锚点会制造成片误报，**必须先修**。

**修法**：命中的 `metadata.heading_path` 已随 JSON 返回（rag-retriever 把 chunk 的
`heading_path` 写进 per-chunk metadata）。`enrich_hit` 取默认片段前：若 `text` 以
`heading_path + "\n\n"` 开头则剥掉该前缀（`startswith` 防御式判断，不匹配则不动）。
只影响锚点片段；`text` 本身保留面包屑供 agent 读上下文。**不动 rag-retriever 本体**
——面包屑是好设计。

**回归测试**（`test_rag.py`）：构造带面包屑前缀的命中（含"标题间有正文"的典型判决书
场景），默认锚点必须过真实 `lint.scan_case` 0 违规；无 `heading_path` / 前缀不匹配 /
frontmatter 首 chunk 等既有路径不回归。

## 2. 后闸门：`lint.py answer` 子命令（核心）

```
python <SKILL_DIR>/lint/lint.py answer <案件根目录> <回答草稿.md>
```

对回答草稿做三项确定性检查（严格度档位经用户拍板：**锚点全验 + 整篇兜底，零误报**——
不猜"哪句话是事实陈述"，那是蕴含判官的活）：

1. **锚点全验**：草稿里出现的每个锚点 `〔来源: …：「…」〕`，逐字片段（经既有归一化）
   确在所指源文件。复用 `_check_anchors`——它本就接受 `(路径, 标识, 正文)` 页面列表，
   草稿当一页喂入即可。
2. **闭世界**：每个锚点的来源路径必须以 `_md/` 开头（分隔符归一后判断）。引用证据
   宇宙之外的东西直接违规。**仅 answer 模式启用**，不追溯既有 wiki（`check` 不变）。
3. **整篇兜底**：若全篇零有效锚点，必须满足其一才放行：
   - 含「未在本案材料中找到」原话（`qa.md` 既有的规范措辞，字面匹配）；
   - 或全部实质内容都在 `> ` 开头的 callout 行 / 标题行 / 空行内（纯分析回答）。
   否则违规——整篇裸答被打回。

退出码与 `check` 同构：0 通过 / 1 违规 / 2 用法错。输出沿用 `check` 的报告风格。

**误报率设计为 ~0**：问候语、过渡句、综合性结论都不会被抓；agent 想陈述事实就必须
挂锚点，挂了就会被验。

## 3. 前闸门：`tools/evidence.py`（新文件，组合既有工具）

```
python <SKILL_DIR>/tools/evidence.py <案件根目录> "<问题>" [--terms "50万,张三,第八条"] [-k 8]
```

一条命令代码执行完三路取证，输出统一 JSON 证据包（把 `qa.md` 现在的三段 prose
取证指令压成一条命令——漏路的可能性从"忘了三条 prose"降到"忘了一条命令"）：

- **RAG 路**：同目录 `import rag` 调 `search_case`（`rag.py` 仍是 rag-retriever 的
  唯一边界，evidence 不直接碰 CLI），命中自带现成锚点与 quality。
- **精确词路**：对 `--terms` 的每个词（agent 从问题里抽出金额/人名/法条号/案号传入）
  纯 Python 精确子串扫 `_md/**/*.md`，命中带前后上下文与**现成锚点**（取命中行逐字拼，
  天然过 lint）。中文精确子串无需分词，零依赖。
- **结构路**：附每份 `_md/` 的标题树（outline 逻辑），对抗"问题措辞与原文用词不同"
  的向量漏检。

**降级即兜底**：rag-retriever 未装/未建索引/模型不一致时，`rag` 字段带
`rag_available:false` + 原因，grep 与 outline 照常返回——证据包永不空手，
降级路径比现状（prose 里"仍不行才退仅 wiki"）更强。

**wiki 路刻意不进这条命令**：读 `index.md`、顺 `[[wikilink]]`、`graph.py` 多跳，
是 agent 自由导航的部分——保留 LLMwiki 式人类思考，正是本设计"中段放开"的含义。

## 4. 协议改动：`qa.md` + `SKILL.md`

- **取证步**：多路取证改为"必跑 `evidence.py` 一条命令（RAG/精确词/结构三路）+
  wiki 路自由导航（index → wikilink → graph.py）"。四情形分流逻辑不变。
- **交付步（新增铁规）**：回答先写草稿文件 → 跑 `lint.py answer` → **0 违规才允许
  发给用户**；违规则有界修复后重跑（同蕴含判官的精神：只许把引用改真实、把断言改
  忠实，绝不为过闸门编造锚点）。草稿文件位置不限（agent 自己的临时目录即可），
  路径显式传给 lint。
- **情形④升级（本次并入）**：两边逐字原文并列 + 各自锚点**不变**（安全阀原样保留）；
  新增——agent **应当**附上标注为 `> [!note] 分析` 的倾向：立场 + 支撑理由 +
  自己排除不掉的反方证据 + 「最终由你裁决」。倾向绝不替代或压缩两边原文的完整并列；
  措辞取"请你来反驳我"（摆足理由与反证供人推翻），不取"建议照此办"（诱导盖章）。
  依据：递交裁决权 ≠ 放弃分析；倾向本质是 INFERRED，用现成的三类标注槽位，
  不破坏"绝不编造事实"。
- **`SKILL.md`**：问答小节同步（evidence 命令入口 + answer 闸门铁规），一两句话，
  细节留在 `qa.md`。

## 5. 可选加硬：Claude Code Stop hook（`setup.md` 增补）

skill 生态里唯一能拿到的"代码拓扑"等价物：案件目录 `.claude/settings.json` 配
Stop hook，agent 每次答完自动抽最后一条回复中的锚点跑校验，违规则阻止交付并回注原因。

**诚实的边界**：hook 无法可靠区分"案件问答"与日常对话（"帮我改个文件名"的回答
没有锚点，不该被拦），所以 hook 只执行检查 1 + 2（锚点真实性 + 闭世界，零误报），
**不执行兜底规则**——密度约束靠协议里的 answer lint 全量检查。

Claude Code 专属、opt-in、`setup.md` 文档化；其他 agent（Codex/Copilot）保持
prose + 闸门工具随 skill 走（工具零依赖，任何 Python 可跑）。

## 6. 测试

沿用现有风格：`tools/` 用 stdlib unittest（零依赖、任何 Python 可跑），`lint/` 用
pytest 风格。

- `test_rag.py` 补：面包屑剥离回归（典型判决书场景默认锚点过真实 lint）、
  无 heading_path 不回归、前缀不匹配不误剥。
- `test_lint.py` 补 answer 子命令全分支：好锚点过 / 片段不符抓 / 缺文件抓 /
  非 `_md/` 路径抓（闭世界）/ 整篇裸答抓 / 含「未在本案材料中找到」放行 /
  纯 callout 分析放行 / 退出码语义。
- 新 `test_evidence.py`：RAG 可用与降级两态的证据包结构、grep 命中的现成锚点
  过真实 lint、`--terms` 解析。

## 7. 非目标（本次不做）

- 不改 rag-retriever 本体（面包屑前缀保留）。
- 不做"回答草稿过蕴含判官"——机制现成（`lint.py extract` 的 claim↔引文抽取思路
  可平移到草稿），列入未来的"要害问题模式"。
- 不校验回答里的 `[[wikilink]]`（回填 wiki 时自有 `check` 把关）。
- 不给其他 agent 做 hook 等价物。
- 不动 ingest 流程与 lint `check`/`extract` 的既有行为。

## 8. 落地顺序（每段独立可验）

1. **面包屑修复**（`tools/rag.py` + 回归测试）——answer 闸门的前置。
2. **`lint.py answer` 后闸门**（+ `test_lint.py`）——价值最大、改动最小。
3. **`tools/evidence.py` 前闸门**（+ `test_evidence.py`）。
4. **协议接线**（`qa.md` 取证步/交付步/情形④ + `SKILL.md`）+ **hook 文档**
   （`setup.md`）。
5. 端到端验证：真实/合成案件上走"提问 → evidence → 综合 → answer 闸门"全链路，
   覆盖四情形与 RAG 降级态。
