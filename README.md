# AnyDocsMarked

**把一堆杂乱的案件材料，变成可控、可溯源、经得起交叉验证的案件知识库——再就案情做可回溯的问答。**

为**法律工作**而生：律师、法务、法官助理面对的是成百上千页判决书、合同、笔录、票据、聊天记录，
其中**每一个金额、日期、当事人、法条都不能错**，而且**任何一句结论都要能指回原文**。
AnyDocsMarked 不追求"看起来很懂"，而是把「可信」与「可溯源」做成第一性原则——
**宁可查不出，也不编造**。

完整的用户目标、产品边界与验收标准见 [`docs/product-requirements.md`](docs/product-requirements.md)。

## 设计目的：它解决的真问题

通用 AI 助手和普通 RAG 用在法律场景上，会在最不能出错的地方出错：

| 法律工作的硬要求 | 通用 AI / 普通 RAG 的问题 | AnyDocsMarked 的做法 |
|---|---|---|
| 结论必须能指回原文 | 生成式回答无法逐字溯源，易"看似有据实则脑补" | 每句事实挂**逐字来源锚点** `〔来源: _md/…：「逐字原文」〕`，**确定性 lint 机器校验** |
| 金额/日期错一位就是事故 | OCR 静默出错、模型幻觉数字 | **双 OCR 互校**（Paddle×MinerU）+ 质检标记 + 数字/日期位专项比对 |
| 近义法律术语必须分清 | 纯向量检索把"表见代理/无权代理"混为一谈 | **BM25 关键词 + 向量混合检索**（RRF 融合），术语精确召回 |
| 推断不能冒充事实 | 模型把分析当事实陈述 | **三类标注铁律**：原文 / 推断 / 存疑物理隔离，越界即拦 |
| 只答本案，不答"一般来说" | 模型用通用法律知识填补本案空白 | **闭世界问答铁规**：答前必检索，锚点必须指向本案材料，裸答被机器打回 |
| 敏感案卷不能随便外传 | 云服务默认上传 | **可完全本地离线、默认不外发**；OCR、标题 LLM 等外部处理需**显式同意** |
| 漏检等于没做 | 一次检索命不中就当没有；漏 ingest 无人知晓 | **三层互补** + **wiki×RAG 交叉验证** + **三态覆盖账本**（每份源文件要么被引用、要么显式登记跳过） |

## 设计哲学

这些原则贯穿三个协作模块的每一处取舍，理解了它们就理解了整个仓库：

### 1. 可信 > 聪明；宁可查不出，也不编造

知识库的价值上限是"聪明"，下限是"可信"——法律场景里下限决定一切。
所有设计都优先保下限：查不到就明说「未在本案材料中找到」，绝不用通用知识填补冒充本案事实；
OCR 互校失败也绝不丢转换结果，只是打上可疑标记随文件进入下游。

### 2. 确定性闸门优先于模型判断

能用规则机器校验的，一律用确定性代码把关，不依赖模型"自觉"：
锚点逐字存在、`[[]]` 死链、时间线日期顺序、勾稽算术、源文件覆盖率、闭世界越界——
全部由**零第三方依赖的 lint** 卡住，退出码非 0 就不放行。
模型只负责确定性代码做不了的部分（归纳、综合、蕴含判断），且判断结果仍受闸门约束。

### 3. 锁确定性两端，中段保持 agent 自由

不做固定管线、不做独立界面。取证入口有**前闸门**（`evidence.py` 一条命令跑齐多路证据），
交付出口有**后闸门**（`lint answer` 0 违规才许发）——两端确定性锁死，
中间的推理、导航、组织答案完全交给 agent 自由发挥。约束住"进"和"出"，就不必约束"想"。

### 4. 倾向 ≠ 裁决：递交裁决权不等于放弃分析

wiki 与 RAG 证据矛盾且查不出原因时，agent **必须**把两套答案 + 各自锚点并列交用户裁决（安全阀），
但**应当**附上明确标注为分析的倾向：立场 + 理由 + **排除不掉的反方证据** + 「最终由你裁决」。
措辞取"请你来反驳我"，不取"建议照此办"。机器永远不静默取舍，人永远握有最终裁决权。

### 5. 松耦合 + 稳定契约

三个协作模块**互不 import**，仅通过 CLI / JSON / frontmatter 契约衔接
（如 `source`、`quality: suspect`、`report.json`）。任一环可单独使用、单独替换、单独测试。

### 6. 可降级 + 默认隐私

lawiki 核心零第三方依赖，只要有 Python 就能跑校验；RAG 与 OCR 是可插拔外部能力，
没装不阻塞核心，问答自动降级并明确告知用户"当前无 RAG 交叉验证"。
默认不外发；选择云端 OCR 或标题 LLM 等外部处理仍需 `--cloud-consent` 显式同意，**非交互也绝不静默上传**。

## 仓库构成

三个**各自独立、通过稳定契约协作、互不 import** 的模块：

| 子项目 | 职责 | 形态 |
|---|---|---|
| [`makeitdown/`](makeitdown) | 把各式文件（PDF / Word / 扫描件 / 图片 / 老式 .doc/.wps…）转成**带质量标记**的 LLM 可读 markdown | CLI（云端 OCR 默认 + 本地可选，双 OCR 互校可选） |
| [`rag-retriever/`](rag-retriever) | 本地优先的语义检索引擎：**结构感知分块 + BM25/向量混合检索**，不含 LLM，只做 RAG 前半程 | CLI / MCP 工具 |
| [`lawiki/`](lawiki) | 法律案件 wiki 构建 skill：把 `_md` 归档成**可溯源**的案件 wiki（案件主体 / 法律关系 / 法律事实 / 时间线），并做 **wiki×RAG 交叉验证问答** | agent skill（确定性 lint + 取证/图谱/outline 工具，核心零依赖） |

各模块保留各自的 README 与许可证，可独立使用：
[makeitdown](makeitdown/README.md) · [rag-retriever](rag-retriever/README.md) · [lawiki](lawiki/README.md)

## 主要架构与数据流

```
原始资料/                     不可变的原始案卷（PDF/Word/扫描件…）
   │
   │  makeitdown（转换 + 质检 + 可选双 OCR 互校）
   ▼
_md/                          带 frontmatter 的高保真 markdown（版本化哈希封存来源层）
   │                          每份记录 provenance_version / source / engine / 双 SHA-256 / 质量标记
   ├─ lawiki ingest ───────▶  wiki/   案件主体·法律关系·法律事实·时间线（每句挂逐字锚点）
   │                                  ingest 收尾自动过 lint + 换实例蕴含判官
   └─ rag-retriever index ─▶  .rag/   结构感知分块 + 向量 + BM25 全文索引
                                 │
      问答时：lawiki（综合结论）  ◄── wiki×RAG 交叉验证 ──►  rag-retriever（原文召回）
                                 一致则答；不一致 → 暴露给人裁决
                                 回答草稿过 lint answer 闸门，0 违规才交付
```

**三层互补，各司其职、互为校验：**

- **`_md`（原文层）**：makeitdown 的忠实转换；原件与正文 SHA-256 由 lint 复核，修改会显式报错。
- **`wiki`（综合层）**：lawiki 归纳出的结构化结论，每句挂逐字锚点、可机器校验。
- **`.rag`（召回层）**：rag-retriever 的混合检索，防漏检的安全网，可重建、可选。

三层**独立**产生，问答时 **wiki×RAG 交叉验证**：wiki 是"人类式思考"（主检索、主作答），
RAG 是"机器式"（独立检索、用于验证）——两条独立路径得同一答案才可信，不一致即报警。
RAG 独立检索必须保留，因为它抓得住 wiki 归纳时的遗漏。

## 技术创新点

### 1. 机器可校验的逐字溯源（护城河）

wiki 每句作为事实的陈述必须挂 `〔来源: _md/…：「逐字原文」〕`，`lawiki/lint` 确定性核验：
锚点逐字确在源文件、`[[]]` 无死链、时间线日期非递减、`> [!check] a+b==c` 勾稽算术成立
（安全求值不 eval）。校验只消除格式噪声（空白/换行/全半角标点/千分位逗号/markdown 标记），
数字与文字精确比对——**"数字写错/张冠李戴"必被抓，"换行/标点差异"不误报**，测试锁死该边界。

### 2. 三类标注铁律 + 换实例蕴含判官

写进 wiki 的每句话必归入 **EXTRACTED（原文直取）/ INFERRED（推断）/ AMBIGUOUS（存疑）**
之一并打标，无法归类者不写。每次 ingest 收尾，`lint extract` 把每条（断言, 引文, 来源, 上下文）
拆出，交**换实例的 LLM 判官**三分判"引文是否支持断言"——专抓"引文是真的、但断言被拔高/
脑补/歪曲"这类锚点校验抓不到的失真。判官只判不改；修复有界（只许把断言改忠实，绝不编造），
三轮仍不过则显著上报用户。

### 3. 问答双闸门：evidence 前闸门 + answer 后闸门

- **前闸门**：`tools/evidence.py` 一条命令跑齐三路取证——RAG 语义检索 + 精确词 grep
  （金额/姓名/法条号这类向量常漏的硬词，归一化匹配）+ `_md` 标题树 outline（措辞不同时按结构导航）。
  各路互补不同的漏检模式，证据包绝不空手：RAG 不可用时其余两路照常返回。
- **后闸门**：回答发给用户前必须过 `lint answer`——确定性校验每个锚点逐字属实、
  全部指向**本案** `_md/`，并逐行拒绝未挂锚点的事实；只有明确的分析 callout 可豁免。
  **裸答直接打回**；反复不过时把违规原样告知用户，绝不静默交付。

### 4. 三态覆盖账本：漏检不再静默

ingest 的完成标准是机器可判的：`lint check` 把 `_md/` 下每份源文件判为
**已引用 / 已登记跳过 / 未处置**三态。决定跳过的文件（草稿、重复扫描版等）必须在 `log.md`
写 skip 条目并附原因，不许静默降格；**未处置 > 0 就不许宣称 ingest 完成**。
"漏了一份关键证据没入库"从此是机器报警，不是运气。

### 5. 双 OCR 互校，专治法律高危盲点

`--ocr-cross-check` 用 **Paddle + MinerU 两个独立引擎**在同一张（先摆正的）页面上各识别一次，
归一化后比对，**重点盯金额/日期/数字位不一致**——OCR 改一位最致命处。分歧标 `quality: suspect`
随文件进入知识库，下游引用自动带「（未核验）」后缀；互校失败也绝不丢转换结果。

### 6. 结构感知分块 + 混合检索

rag-retriever 按**法律文书结构**切块：沿 markdown 标题切分（每块带"出自哪一节"的面包屑）、
表格不切开、`第X条` / `本院认为`等法律标记做软边界。检索默认 **BM25 关键词（jieba 分词，
全离线）+ 向量双路 RRF 融合**——纯向量会把"表见代理 vs 无权代理"混淆，关键词通道把术语
当硬约束。可选本地 cross-encoder 重排（`bge-reranker-v2-m3`，默认关）。

### 7. LLM 标题重建：原理上碰不到正文

扫描件 OCR 出来是扁平文本。可选的 `--structure-headings` 用 LLM 重建标题层级，但 LLM
**只返回"行号 → 标题级别"的数字**，绝不经手正文——加 `#` 由本地完成，正文逐字节原样保留。
金额/日期/当事人**在原理上零篡改风险**，任何失败都回退原文。

### 8. 确定性导航工具，替代人肉追链

- `tools/graph.py`：只走 wiki 中已过 lint 的 `[[wikilink]]` 建图，回答"X 与 Y 有无关系"
  这类多跳问题（neighbors / 最短路径），零依赖、结果可复现。
- `tools/outline.py`：`_md` 全库标题树，对抗"问法与原文用词不同"的漏检，亦作无 RAG 时的降级通道。

### 9. 国内可用 + 真离线

- Release 提供 `-offline.zip`（内置 embedding ONNX 模型 + tiktoken 词表），解压即离线可用，
  CI 发布时机器校验包内确实含模型。
- `install.py --check-offline` 自检**实际运行时实例**能否离线出向量（不是看文件"像不像"齐了）。
- 依赖走阿里云 PyPI 镜像、MinerU 权重默认 ModelScope、代码有 Gitee 镜像——全链路不依赖海外网络。

## 使用方式

### 完整流程

1. **建案件目录**，把原始文件放进 `原始资料/`。
2. **转换**：`makeitdown 原始资料 -o _md`——递归转换、自动质检；扫描件走 OCR，
   法律高危件可加 `--ocr-cross-check` 双 OCR 互校。批量大时后台运行，逐文件进度打到 stderr，
   中断后 `--skip-existing` 断点续传，完成以 `report.json` 为准。
3. **建库**：
   - lawiki skill 读 `_md`，按铁律归档进 `wiki/`（每句事实挂逐字锚点；
     收尾自动跑 lint + 蕴含判官 + 覆盖率清账）。
   - rag-retriever 对 `_md` 建结构感知分块 + 向量/BM25 索引（可选，没装不阻塞）。
4. **问答**：对 agent 说「问本案…」。agent 走闭世界协议：先多路取证，wiki×RAG 交叉验证，
   给出**带逐字来源**的回答；冲突时两版并列 + 附标注为分析的倾向，交人裁决；
   回答过 `lint answer` 闸门才交付。

### 对非技术用户

把整个仓库（或 Release bundle）交给 AI 助手（Claude Code / Copilot 等），说一句
「**整理案件资料**」——助手会自动安装环境、跑完转换与建库；之后直接问案情即可。

### 问答长什么样

- 「本案里 X 公司总共付了多少货款？」→ 回答带 `〔来源: _md/银行流水.md：「…」〕` 逐字锚点，
  wiki 结论与 RAG 原文互证。
- wiki 说 50 万、原文说 45 万且查不出原因 → 两套答案 + 各自锚点并列，附 agent 的倾向分析
  与反方证据，明说「请人工溯源裁决」。
- 材料里根本没有 → 「未在本案材料中找到」，绝不拿通用法律知识充数。

## 安装

### 一键上手（推荐：下载 Release bundle）

到 [Releases](https://github.com/Tsinglaw/AnyDocsMarked/releases) 下载：

- `anydocsmarked-v*-offline.zip` —— 内置 embedding 模型，解压即离线可用，**国内推荐**；
- `anydocsmarked-v*.zip` —— 源码包，首次建索引需联网下模型。

解压后让 AI agent 加载 `skill/lawiki`，按提示跑 `python install.py` 即自动安装
makeitdown 与 rag-retriever（装完可 `python install.py --check-offline` 自检离线就绪）。
然后把文件放进案件目录的 `原始资料/`，对 agent 说「整理案件资料」。

### 从源码安装

两个可安装包与一个无需安装的 skill：

```bash
uv tool install --python 3.12 "makeitdown[local] @ git+https://github.com/Tsinglaw/AnyDocsMarked.git#subdirectory=makeitdown"
uv tool install --python 3.12 "rag-retriever @ git+https://github.com/Tsinglaw/AnyDocsMarked.git#subdirectory=rag-retriever"
```

lawiki 是 skill、无需安装（加载 `lawiki/skill/lawiki` 即可）。
国内镜像加速与更多选项见 [`lawiki/skill/lawiki/references/setup.md`](lawiki/skill/lawiki/references/setup.md)。
注意：git 源码路径装不出离线版（vendor 的模型不进 git），要真离线请用 `-offline` bundle。

## 工程质量

- **CI 全量测试**：三个子项目各自的测试套件在 **Ubuntu + Windows 双矩阵**上跑
  （Windows 是律师桌面的主平台——COM 转换、GBK 控制台、路径分隔符只在那里坏）；
  仓库级 ruff 门禁 + 生产代码分支覆盖率 80% 硬门槛（当前各模块 84%–86%）。
- **lint 自身有测试锁边界**：锚点归一化"抓真错、不误报"的行为由 `test_lint.py` 的
  嵌套引号、千分位、换行、逐行裸答、路径穿越与来源哈希用例钉死。
- **发布即校验**：release 工作流机器验证 offline 包内确实含 ONNX 模型，缺了直接 fail。
- **规格驱动**：每个较大特性先写 spec 再写 plan 再实现，全部留档于
  [`docs/superpowers/`](docs/superpowers)。
- **维护与披露**：版本变化见 [`CHANGELOG.md`](CHANGELOG.md)，敏感漏洞与案卷数据处理基线见 [`SECURITY.md`](SECURITY.md)。
