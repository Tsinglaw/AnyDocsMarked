# 案件问答（交叉验证：wiki 结论 × RAG 原文）

> 用户就**已建好的案件**提问时走本协议。核心：同时取 wiki 已综合结论与 RAG 原文证据——**一致则答；不一致则查因；查不出则把两套答案 + 各自原始依据并列交用户裁决**。继承三类标注纪律（EXTRACTED / INFERRED / AMBIGUOUS）。
>
> RAG 是可降级旁路：取证经 `<SKILL_DIR>/tools/evidence.py` 一条命令（内部经 `tools/rag.py` 调 RAG，见 `rag.md`）。没装 / 没建索引 / 模型不一致 → 证据包中 `rag_available:false`，但 grep 与 outline 两路照常返回；须明确告知用户「当前无 RAG 交叉验证」。

## 铁规 · 闭世界接地（先检索，后作答）

**回答本案问题前，先读这三条，违反即出错：**

1. **唯一证据宇宙 = 本案 `原始资料/` `_md/` `wiki/` `.rag/`。** 本案的任何事实**只能**来自这些文件检索所得；**不在其中的，不是本案事实**。
2. **先检索，后作答。严禁凭记忆 / 通用法律知识直接回答本案问题。** 必须先走「多路取证」（下节），用检索到的原文支撑，每个事实陈述挂逐字锚点 `〔来源: _md/…：「原文」〕`。
3. **检索不到就直说「未在本案材料中找到」，绝不用通用知识填补冒充本案事实。** 通用法律分析只能进 `> [!note] 分析` 并标明「非本案证据、仅供参考」，与事实物理隔离。

> 一句话：**你的事实只有这一闭集；答前必检索；查无即明说，绝不脑补。**

## 触发

「问本案…」「关于这个案子…」「本案里 X 是什么/是多少/是谁」等**就案件内容提问**——与「整理资料 / 建库 / ingest」这类**构建**触发词区分。构建走 SKILL.md 流水线，提问走本协议。

## 第一步 · 多路并行取证

**各路互补不同漏检，不可只凭一边。**

1. **wiki 路**：读 `wiki/index.md` → 顺 `[[wikilink]]`/grep 找相关页 → 取其**已综合结论 + 既有锚点**。
   - **关系/多跳问题**（"X 与 Y 有无关系""X 牵涉哪些事实/关系"）用确定性图工具，替代人肉追链：
     ```
     python <SKILL_DIR>/tools/graph.py <案件根> neighbors "<页名>"
     python <SKILL_DIR>/tools/graph.py <案件根> path "<A>" "<B>"
     ```
     只走 wiki 中已存在、经 lint 校验的 `[[wikilink]]`（导航页/时间线不入图），返回邻居或最短连通路径；拿到路径后仍读沿途页锚点取证。零依赖、始终可用。
2. **证据包路（一条命令，三路代码执行）**：
   ```
   python <SKILL_DIR>/tools/evidence.py <案件根> "<问题>" --terms "<金额,姓名,法条号,案号>" -k 8
   ```
   先从问题里抽出**精确词**（金额/当事人姓名/法条号/案号——向量按语义检索常漏的）填进 `--terms`；金额写成不带千分位逗号的形式（如 `50000`），匹配按归一化做、照样命中原文的「50,000」。返回统一 JSON：
   - `rag`：语义检索命中（相对路径 + 逐字 `text` + `score` + `quality` + 现成 `anchor`）。`k≥8` 防假冲突：先凑够上下文再判，别把「RAG 没检索到」误判成「矛盾」。`rag_available:false` 时其余两路照常返回——证据包绝不空手，但须告知用户「当前无 RAG 交叉验证」。
   - `grep`：各精确词在 `_md/` 的逐行命中（带现成 `anchor`；来源可疑的带「（未核验）」）。**一个词向量没召回 ≠ 不存在；grep 结果为准**——查无的词列在 `not_found`，这时才可以放心说「未找到」。
   - `outline`：每份 `_md/` 的标题树。**当问题措辞与原文用词不同**（如问「违约责任」而合同写「第八条 责任」），按结构导航到相关文件的相关章节，读该节原文取证、拼锚点。

## 第二步 · 比对，四情形分流

把 wiki 结论与 RAG 逐字原文对齐，落入四者之一：

- **① 一致** → 直接回答；**同时**引用 wiki 页（`[[页名]]`）与 `_md/` 锚点，标注「wiki 与原文互证」。
- **② wiki 沉默 / 未覆盖**（≠冲突）→ 用 RAG 原文回答，挂 `_md/` 锚点，标 `> [!note] 分析`「该点尚未入 wiki」；可建议用户是否 ingest 进 wiki。
- **③ 不一致，能定因** → 查因三连：
  1. **wiki 总结失真？** 比对 wiki 断言 vs RAG 逐字原文（拔高/转述/张冠李戴）。
  2. **RAG 漏上下文？** 提高 `-k` 或换检索词再查。
  3. **源文件本身冲突？** 看 wiki 是否已有 `> [!warning] ⚠ 冲突` callout。
  
  定因后**以 EXTRACTED 逐字原文为准**作答，并指出 **wiki 何处需修**（交用户确认后才回填，见 SKILL.md）。
- **④ 不一致，查不出因**（安全阀）→ **不许静默取舍**：把【wiki 答案 + 锚点】与【RAG 答案 + `_md/` 锚点】**并列**，各附相对路径 + 逐字片段，明说「**无法判定，请人工溯源裁决**」→ 交用户。
  并列之后，**应当**附上你的倾向（`> [!note] 分析` 标注，属 INFERRED）：倾向哪版 + 支撑理由 + **你排除不掉的反方证据** + 「最终由你裁决」。措辞取"请你来反驳我"（摆足理由与反证供人推翻），不取"建议照此办"（诱导盖章）。倾向**绝不替代、绝不压缩**两边原文的完整并列——递交裁决权 ≠ 放弃分析。

## 第三步 · 引用纪律（继承铁律）

- 每个事实陈述挂 `〔来源: _md/…：「逐字原文」〕`；逐字片段取自 RAG 返回的 `text`（或直接用 `anchor`，通常缩到具体支撑句）。**一条断言 ← 一条能完整支持它的引文。**
- 来源 `quality: suspect` 的（wrapper 已在 `anchor` 后缀「（未核验）」/置 `unverified:true`）→ 引用时保留该标注，绝不当既定事实。
- 推断标 `> [!note] 分析`；冲突/存疑标 AMBIGUOUS（`> [!warning] ⚠ 冲突` 并列各方 + 锚点）。
- **「一致/不一致」是 LLM 判断、会错** → 永远把两边逐字原文摆出供用户复核，不要只给「一致」二字。
- 问答引用机器可校验：用户认可的问答结论已是 wiki 锚点格式，可直接回填 wiki，并能过 `lint check`。

## 第四步 · 交付闸门（确定性，必过）

回答**发给用户前**：写入草稿文件（任意临时路径均可）→ 跑

```
python <SKILL_DIR>/lint/lint.py answer <案件根> <草稿.md>
```

→ **0 违规（退出码 0）才允许交付**。它确定性校验三件事：① 每个锚点的逐字片段确在所指源文件 ② 解析后的锚点真实位于本案 `_md/` ③ 每个分析 callout 之外的实质内容行都必须挂锚点，或明确写「未在本案材料中找到」——一个合法锚点不能替其他裸事实免责，普通 `>` 引用块也不算分析。

违规 → **有界修复**：只许把引用改真实、把断言改忠实，**绝不为过闸门编造锚点**（同蕴含判官的纪律）→ 重跑；反复不过时把违规原样告知用户，绝不静默交付。

## 共同盲区与降级

- wiki 与 RAG 都从 `_md/` 派生，**抓不出 `_md/` 自身的 OCR 错** → 靠 `quality: suspect` 标注 + 必要时回溯 `原始资料/` 原件兜底。
- **仅 wiki 降级**：RAG 不可用时，照常用 wiki 作答并挂既有锚点，但须对用户挑明「**当前无 RAG 交叉验证，结论仅来自已综合的 wiki**」，建议其装好 RAG / 重建索引后复核要害点。
